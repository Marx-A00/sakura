      - `StorageMode.LOCAL` -> "Just me (local storage)"

    - Add a "Change" button that triggers a migration confirmation dialog:
      - Show an `AlertDialog` with the target mode name and a warning about what happens:
        - Local -> Syncthing: "Your org files will be copied to your Syncthing folder. You'll need to grant storage access and choose a folder."
        - Syncthing -> Local: "Your org files will be copied to local storage on this device."
      - "Cancel" dismisses; "Continue" triggers migration

    **2. Implement migration logic as a suspend function:**
    - Can be a private function in MacroTargetsScreen or extracted to a helper:
    - `suspend fun migrateStorage(context: Context, prefsRepo: AppPreferencesRepository, targetMode: StorageMode)`
    - If migrating Local -> Syncthing:
      - List all `.org` files in `context.filesDir`
      - Get the sync folder path from `prefsRepo.syncFolderPath.first()`
      - If sync folder path is null, the user will need to go through folder selection (similar to onboarding Syncthing path) â€” for v1, show a message that they need to set up sync first
      - Copy each file to the sync folder
      - Call `prefsRepo.setStorageMode(StorageMode.SYNCTHING)`
    - If migrating Syncthing -> Local:
      - Get the sync folder path from `prefsRepo.syncFolderPath.first()`
      - List all `.org` files in the sync folder
      - Copy each to `context.filesDir`
      - Call `prefsRepo.setStorageMode(StorageMode.LOCAL)`
    - After migration, the activity needs to restart for the new AppContainer to be created with the new mode. Use `(context as? Activity)?.recreate()` or show a snackbar suggesting restart. The simplest approach: since `MainActivity` uses `remember(resolvedMode)`, the storageMode DataStore change will trigger recomposition and a new AppContainer â€” BUT this only works if the storageMode state is being collected. The cleanest v1: after `setStorageMode()`, call `(context as ComponentActivity).recreate()` to force a full restart.

    **3. For Local -> Syncthing migration (requires permission/folder that may not exist):**
    - If `syncFolderPath` is null or permission not granted, show a dialog: "To switch to Sync mode, you'll need to grant storage access. This will restart the app." Then reset onboarding: `prefsRepo.setStorageMode(StorageMode.SYNCTHING)` and clear onboarding complete. On recreate, the app will show onboarding flow which now starts at ModeSelection â€” but since mode is already SYNCTHING, it should go to permission check. HOWEVER the onboarding flow starts at Welcome. A simpler approach: set storageMode to SYNCTHING but DON'T clear onboarding. Instead, add a route/screen for Syncthing setup that can be navigated to from Settings. For v1 simplicity: only allow Syncthing -> Local migration in the Settings UI. Local -> Syncthing is a power-user operation that can be added later. ACTUALLY, re-reading CONTEXT.md: "Bidirectional switching supported." So implement both, but keep it simple:
      - Syncthing -> Local: straightforward copy + mode switch + recreate
      - Local -> Syncthing: set mode to SYNCTHING, clear onboarding flag, recreate â€” this forces the user through the Syncthing onboarding flow (Welcome -> ModeSelection -> Permission -> Folder). Since they already chose Syncthing, the onboarding should detect the stored mode and skip to the permission/folder screens. Add logic in OnboardingViewModel: if `prefsRepo.storageMode.first() == SYNCTHING` in the init block, skip Welcome and ModeSelection, go directly to `checkPermission()`.

    **4. Update `OnboardingViewModel.kt` for re-onboarding:**
    - In init (or a new `initialize()` method called from init), check if storageMode is already set:
      ```
      viewModelScope.launch {
          val existingMode = prefsRepo.storageMode.first()
          if (existingMode == StorageMode.SYNCTHING) {
              // Re-onboarding for Syncthing setup â€” skip Welcome/ModeSelection

          }
          // else: stay on Welcome (new user or re-onboarding for local which shouldn't happen)
      }
      ```
    - This handles the case where a local-mode user switches to Syncthing via Settings

    **5. Wire Settings screen with context for migration:**

    - For `recreate()`, get the Activity from context: `(context as? ComponentActivity)?.recreate()`
